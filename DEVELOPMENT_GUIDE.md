# Development Guide: Architecture & Patterns

## 1. Core Principles
- **Universal User ID**: The UUID generated by `auth-service` is the single identifier for users across ALL services.
- **Contract-First**: API contracts must be defined in `libs/openapi-specs` before implementation.
- **Domain-Driven Design (DDD)**: Logic is organized into Domain, Application, and Infrastructure layers.
- **Microservices Integrity**: Services are independent but communicate via Kafka events for eventual consistency.

## 2. Backend Architecture (NestJS)

### Layered Structure
All services MUST follow this DDD pattern within `src/`:
- **`domain/`**: Entities, Value Objects, Domain Services, and Domain Events. (Pure logic, no external dependencies).
- **`application/`**:
  - `use-cases/`: The entry point for logic. (Calls domain logic and infra ports).
  - `ports/`: Interfaces for repositories, messaging, and external services.
  - `dto/`: Data Transfer Objects for requests/responses.
- **`infrastructure/`**:
  - `persistence/`: Prisma/Mongoose service and repositories.
  - `messaging/`: Kafka producer/consumer implementations.
  - `cache/`: Redis implementation.
  - `strategies/`: Passport strategies (JWT, Google).
- **`interfaces/`**:
  - `controllers/`: HTTP endpoints.
  - `mappers/`: Transformation between Domain/DSO and API DTOs.

### Error Handling
- Use the `shared-exceptions` library for consistent API error responses.
- Never throw raw errors; always wrap them in `HttpException` or a custom shared exception.

### Database & Prisma
- **Multiple Schemas**: Each service has its own Prisma schema in `apps/[service-name]/prisma/schema.prisma`.
- **Generation**: To avoid conflicts in the shared `node_modules`, each schema must generate to a unique location:
  - `auth-service`: `output = "../../../node_modules/@prisma/client-auth"`
  - `user-service`: `output = "../../../node_modules/@prisma/client-user"`
- **Imports**: `PrismaService` must import `PrismaClient` from its specific generated location (e.g., `@prisma/client-auth`).

## 3. Frontend Architecture (Next.js)

### Feature-Based Organization
- Features live in `apps/frontend/src/features/[feature-name]/`.
- **Strict Typing**: Never use `any` in frontend code.
  - Define explicit interfaces for API responses and component props.
  - Use `unknown` with type guards or `instanceof` checks for error handling.
- Structure per feature:
  - `components/`: UI components specific to the feature.
  - `hooks/`: Custom React hooks.
  - `services/`: API client calls (Axios/Fetch).
  - `store/`: Zustand/Redux state slices.

### Authentication Pattern
- **HttpOnly Cookies**: JWTs are managed by the browser automatically.
- **Data Fetching**: Use **TanStack Query (React Query)** for all API requests.
  - Use `useQuery` for fetching data (e.g., profiles).
  - Use `useMutation` for actions (e.g., login, signup, logout).
- **State**: The `authStore` only stores user profile data and `isAuthenticated` status.
- **Requests**: All API calls must include `credentials: 'include'`.
- **Interceptors**: Use a global interceptor for silent token refresh on 401 errors.

## 4. Coding Standards
- **Naming**: `kebab-case` for files and folders. `PascalCase` for classes/components. `camelCase` for functions/variables.
- **Strict Typing**: Always provide explicit TypeScript types.
  - Avoid `any` at all costs across ALL files (source code, tests, and tools).
  - Use `unknown` with type guards or `instanceof` checks for dynamic data.
- **Linting**: Follow `.eslintrc.json` rules strictly.
  - No `/* eslint-disable */` is permitted. Fix the root cause or use `unknown as T` for necessary casting in tests.
  - CI/CD will fail on ANY linting error.
- **Security**: Never log sensitive data (passwords, tokens). Use `class-transformer` to exclude sensitive fields from responses.

## 5. Deployment & Environment
- **Env**: Use Zod-based validation for all environment variables (`env.validation.ts`).
- **Compose**: Local development is powered by the root `docker-compose.yml`.
